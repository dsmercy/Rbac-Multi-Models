# 🎯 Context

The admin panel is the management UI for an enterprise RBAC system that supports:

* **Multi-tenancy** — each tenant is an isolated organisation; the current tenant is always in scope
* **Roles & permissions** — roles group permissions; users are assigned roles within a scope hierarchy
* **Scoped access** — org → department → project hierarchy with downward inheritance
* **ABAC policies** — JSON condition trees evaluated at runtime
* **Delegation** — time-bound, subset-only access grants between users
* **Audit logging** — immutable, append-only log of every access decision and admin action

The backend is a .NET 8 REST API with JWT authentication, Redis caching, and SignalR WebSocket support. All API contracts are versioned under `/api/v1/`.

## Centralised Route Config

All route definitions live in `/src/routes/` — never scattered across feature folders or component files. The router reads a single composed config; features contribute their route segments by exporting a config object, not by rendering `<Route>` elements themselves.

### Why centralised

* One place to audit every URL in the app
* Lazy loading is configured at the route level, not inside components
* Tenant scope enforcement is applied uniformly in one layout, not repeated per feature
* Adding or removing a feature's routes is a single-file change with no side effects elsewhere

### Route config types

```typescript
// /src/routes/types.ts

export interface AppRoute {
  path: string;
  component: React.LazyExoticComponent<React.ComponentType>;
  // Optional guard — evaluated before the component renders.
  // Return true to allow, false to redirect to /login.
  guard?: (context: RouteGuardContext) => boolean;
}

export interface RouteGuardContext {
  isAuthenticated: boolean;
  tenantId: string | null;
  can: (action: string, resource: string) => boolean;
}
```

### Feature route configs

Each feature exports a plain array of `AppRoute` objects from its `index.ts`. No JSX, no `<Route>` — just data.

### Router root

```typescript

import { Suspense } from 'react';
import { TenantLayout } from '@/shared/components/TenantLayout';
import { AuthGuard } from '@/shared/components/AuthGuard';
import { RouteGuard } from '@/shared/components/RouteGuard';
import { Skeleton } from '@/shared/components/Skeleton';

const NotFoundPage = lazy(() => import('@/shared/components/NotFoundPage'));

const router = createBrowserRouter([
  {
    // Public routes — no auth required
    path: '/',
    element: <Suspense fallback={<Skeleton />}><LoginPage /></Suspense>,
  },
  {
    path: '/auth/callback',
    element: lazy(() => import('@/features/auth/components/OAuthCallback')),
  },

  {
    // All tenant-scoped routes live under /:tenantId
    // TenantLayout validates the tenantId param and sets the ambient tenant context
    path: '/:tenantId',
    element: (
      <AuthGuard>
        <TenantLayout />
      </AuthGuard>
    ),
    children: tenantRoutes.map((route) => ({
      path: route.path,
      element: (
        <RouteGuard guard={route.guard}>
          <Suspense fallback={<Skeleton />}>
            <route.component />
          </Suspense>
        </RouteGuard>
      ),
    })),
  },

  {
    path: '*',
    element: <NotFoundPage />,
  },

  return <RouterProvider router={router} />;

```

### `TenantLayout` responsibilities

`TenantLayout` is the single component that owns tenant context. It must:

1. Extract `:tenantId` from the URL param via `useParams()`
2. Validate the `tenantId` matches the `tid` claim in the current session (call `GET /api/v1/auth/me` which is already cached by RTK Query) — redirect to `/` with a `403` toast if they differ
3. Set the ambient tenant context in the Redux store (`dispatch(setTenantId(tenantId))`) so `buildBaseQuery` can inject `X-Tenant-Id` on every subsequent request
4. Fetch and cache the user's permission set for this tenant via `useCheckPermissionsQuery` — this populates `AbilityContext` for all child routes
5. Render the nav shell (sidebar, top bar, notification bell) and an `<Outlet />` for child routes

A user cannot access `/:tenantId/*` for a tenant they do not belong to — the layout enforces this before any feature component renders.

### `AuthGuard`

```typescript
// /src/shared/components/AuthGuard.tsx

export function AuthGuard({ children }: { children: React.ReactNode }) {
  const { isAuthenticated, isLoading } = useAuthStatus();

  if (isLoading) return <Skeleton />;
  if (!isAuthenticated) return <Navigate to="/" replace />;

  return <>{children}</>;
}
```

### `RouteGuard`

```typescript
// /src/shared/components/RouteGuard.tsx

interface RouteGuardProps {
  guard?: (ctx: RouteGuardContext) => boolean;
  children: React.ReactNode;
}

export function RouteGuard({ guard, children }: RouteGuardProps) {
  const ctx = useRouteGuardContext();   // builds RouteGuardContext from store + AbilityContext

  if (!guard) return <>{children}</>;

  return guard(ctx)
    ? <>{children}</>
    : <Navigate to="../" replace />;    // relative redirect — stays within the tenant segment
}
```

### Lazy loading

Every page component is wrapped in `React.lazy()` at the point of import in the feature's route config. The `Suspense` boundary with a `<Skeleton />` fallback is applied once in the router root — not per feature, not per page. This means:

* Each page's JS bundle is only downloaded when the user first navigates to that route
* Adding a new page never touches the router root — only the feature's route config export
* The loading experience is consistent across all routes

Never import page components eagerly in route configs. The pattern `lazy(() => import('./components/RoleListPage'))` is the only permitted form.

## Module Boundary Rules

* No feature may import directly from another feature's internal files — only from its `index.ts` public exports
* No API calls inside components — all data fetching goes through RTK Query hooks defined in `/shared/api`
* No business logic inside components — extract to custom hooks
* Zustand stores are feature-scoped; no single global god-store
* No `<Route>` elements inside feature components — route structure is defined in route config objects only

# 🔌 Phase 2 — API Layer

## Centralised API Slice

There is exactly **one** `createApi` call in the entire codebase, located in `/src/shared/api/apiSlice.ts`. This is the single source of truth for all server communication. Features never create their own `createApi` instances — they inject endpoints into this shared base.

### Why one slice

Multiple `createApi` instances create isolated caches that cannot share tags, invalidate each other's data, or deduplicate in-flight requests. A single slice means a mutation in one feature can invalidate a query in another without any cross-feature coupling.

### Base slice definition

```typescript
// /src/shared/api/apiSlice.ts

import { createApi } from '@reduxjs/toolkit/query/react';
import { buildBaseQuery } from './buildBaseQuery';

// Tag types are the exhaustive list of cacheable entity types in the app.
// Add a new entry here whenever a new entity is introduced — nowhere else.
const TAG_TYPES = [
  'Role',
  'Permission',
  'User',
  'Policy',
  'Delegation',
  'AuditLog',
  'Tenant',
  'Notification',
] as const;

export type TagType = (typeof TAG_TYPES)[number];

export const apiSlice = createApi({
  reducerPath: 'api',
  baseQuery: buildBaseQuery,        // custom — see Base Query section below
  tagTypes: TAG_TYPES,
  endpoints: () => ({}),            // all endpoints injected per feature
});
```

### Feature endpoint injection

Each feature injects its own endpoints using `apiSlice.injectEndpoints`. The feature file only touches its own tag types — it never reaches into another feature's tags directly.

```typescript
// /src/features/roles/roleEndpoints.ts

import { apiSlice } from '@/shared/api/apiSlice';
import type { Role, CreateRoleInput, UpdateRoleInput } from './types';

export const roleEndpoints = apiSlice.injectEndpoints({
  endpoints: (builder) => ({

    getRoles: builder.query<Role[], { tenantId: string }>({
      query: ({ tenantId }) => `/tenants/${tenantId}/roles`,
      providesTags: (result) =>
        result
          ? [
              ...result.map(({ id }) => ({ type: 'Role' as const, id })),
              { type: 'Role', id: 'LIST' },
            ]
          : [{ type: 'Role', id: 'LIST' }],
    }),

    getRoleById: builder.query<Role, { tenantId: string; roleId: string }>({
      query: ({ tenantId, roleId }) => `/tenants/${tenantId}/roles/${roleId}`,
      providesTags: (_result, _err, { roleId }) => [{ type: 'Role', id: roleId }],
    }),

    createRole: builder.mutation<Role, { tenantId: string; body: CreateRoleInput }>({
      query: ({ tenantId, body }) => ({
        url: `/tenants/${tenantId}/roles`,
        method: 'POST',
        body,
      }),
      invalidatesTags: [{ type: 'Role', id: 'LIST' }],
    }),

    updateRole: builder.mutation<Role, { tenantId: string; roleId: string; body: UpdateRoleInput }>({
      query: ({ tenantId, roleId, body }) => ({
        url: `/tenants/${tenantId}/roles/${roleId}`,
        method: 'PUT',
        body,
      }),
      invalidatesTags: (_result, _err, { roleId }) => [{ type: 'Role', id: roleId }],
    }),

    deleteRole: builder.mutation<void, { tenantId: string; roleId: string }>({
      query: ({ tenantId, roleId }) => ({
        url: `/tenants/${tenantId}/roles/${roleId}`,
        method: 'DELETE',
      }),
      invalidatesTags: (_result, _err, { roleId }) => [
        { type: 'Role', id: roleId },
        { type: 'Role', id: 'LIST' },
      ],
    }),
  }),

  overrideExisting: false,
});

// Export typed hooks — generated automatically by RTK Query
export const {
  useGetRolesQuery,
  useGetRoleByIdQuery,
  useCreateRoleMutation,
  useUpdateRoleMutation,
  useDeleteRoleMutation,
} = roleEndpoints;
```

Apply this same pattern to every feature. Each feature file owns its own endpoint definitions and exports its own hooks. Nothing else in the codebase calls `apiSlice` directly.

### `LIST` tag convention

Every collection query provides a sentinel `{ type: 'X', id: 'LIST' }` tag. Every create mutation invalidates `LIST`. Item-level mutations (update, delete) invalidate the specific `{ type: 'X', id }` tag only — not the whole list — to avoid unnecessary refetches when editing a single record. This is the standard RTK Query pattern; apply it uniformly across all features.

### Adding a new entity

When a new entity is introduced:
1. Add its tag type to `TAG_TYPES` in `apiSlice.ts` — one place, one line
2. Create `/src/features/{entity}/{entity}Endpoints.ts` following the pattern above
3. Export hooks from the feature's `index.ts`

No other file needs to change.

## Custom Base Query (`buildBaseQuery`)

The base query lives in `/src/shared/api/buildBaseQuery.ts` and is the single place for all cross-cutting HTTP concerns. Features never configure these themselves.

```typescript
// /src/shared/api/buildBaseQuery.ts

import {
  fetchBaseQuery,
  type BaseQueryFn,
  type FetchArgs,
  type FetchBaseQueryError,
} from '@reduxjs/toolkit/query/react';
import { v4 as uuidv4 } from 'uuid';
import type { RootState } from '@/app/store';
import { logout } from '@/features/auth/authSlice';

const rawBaseQuery = fetchBaseQuery({
  baseUrl: import.meta.env.VITE_API_BASE_URL,
  // credentials: 'include' sends httpOnly cookies automatically — no manual token handling
  credentials: 'include',
  prepareHeaders: (headers, { getState }) => {
    const tenantId = (getState() as RootState).auth.tenantId;
    if (tenantId) {
      headers.set('X-Tenant-Id', tenantId);
    }
    // Attach a unique correlation ID to every request for distributed tracing
    headers.set('X-Correlation-Id', uuidv4());
    return headers;
  },
});

let isRefreshing = false;

export const buildBaseQuery: BaseQueryFn<
  string | FetchArgs,
  unknown,
  FetchBaseQueryError
> = async (args, api, extraOptions) => {
  let result = await rawBaseQuery(args, api, extraOptions);

  if (result.error?.status === 401 && !isRefreshing) {
    isRefreshing = true;
    // Attempt a silent token refresh using the httpOnly refresh cookie
    const refreshResult = await rawBaseQuery(
      { url: '/auth/refresh', method: 'POST' },
      api,
      extraOptions,
    );
    isRefreshing = false;

    if (refreshResult.data) {
      // Refresh succeeded — retry the original request
      result = await rawBaseQuery(args, api, extraOptions);
    } else {
      // Refresh failed — session is dead, force logout
      api.dispatch(logout());
    }
  }

  return result;
};
```

The three responsibilities of `buildBaseQuery`:

* **Cookie-based auth** — `credentials: 'include'` means the browser attaches the httpOnly access token cookie on every request. No JS code reads or writes the token value directly.
* **Transparent token refresh** — a single `401` triggers one refresh attempt. A guard flag (`isRefreshing`) prevents concurrent refresh races. If refresh fails, `logout()` is dispatched and the app redirects to `/login`.
* **Cross-cutting headers** — tenant ID and correlation ID are injected once here, not duplicated across endpoint definitions.

## Tag Invalidation Strategy

Every mutation must declare its invalidation tags. The rule: invalidate the minimum set of tags needed — never invalidate `'Role'` (all roles) when only one record changed.

| Mutation | Invalidates |
|---|---|
| `createRole` | `{ type: 'Role', id: 'LIST' }` |
| `updateRole` | `{ type: 'Role', id: roleId }` |
| `deleteRole` | `{ type: 'Role', id: roleId }`, `{ type: 'Role', id: 'LIST' }` |
| `assignPermissionToRole` | `{ type: 'Role', id: roleId }`, `{ type: 'Permission', id: 'LIST' }` |
| `assignRoleToUser` | `{ type: 'User', id: userId }`, `{ type: 'Role', id: 'LIST' }` |
| `revokeRoleFromUser` | `{ type: 'User', id: userId }` |
| `createPolicy` | `{ type: 'Policy', id: 'LIST' }` |
| `updatePolicy` | `{ type: 'Policy', id: policyId }` |
| `deletePolicy` | `{ type: 'Policy', id: policyId }`, `{ type: 'Policy', id: 'LIST' }` |
| `createDelegation` | `{ type: 'Delegation', id: 'LIST' }`, `{ type: 'User', id: userId }` |
| `revokeDelegation` | `{ type: 'Delegation', id: delegationId }`, `{ type: 'User', id: userId }` |

Apply this table exhaustively — every mutation endpoint in every feature must have an `invalidatesTags` declaration. A mutation without `invalidatesTags` is a bug.

# 🔒 Phase 3 — Security

## JWT Storage & Token Refresh

* Access token stored in an **httpOnly cookie** (set by the backend with `Secure`, `SameSite=Strict`). JavaScript cannot read it directly — the browser sends it on every same-origin request automatically.
* Refresh token also in an httpOnly cookie with a longer `Max-Age`.
* The React app never stores tokens in `localStorage`, `sessionStorage`, or any JS-accessible variable. Document why (XSS resistance) and include a comment block in the auth slice to prevent future developers from reversing this decision.
* On app boot, call `GET /api/v1/auth/me` to validate the session. A `200` means the cookie is valid and returns the user profile. A `401` means no valid session — redirect to login.
* Token refresh is handled transparently inside `fetchBaseQueryWithAuth` (see Phase 2) — no component ever triggers a refresh manually.

## UI-Level RBAC (Authorization)

Implement an `AbilityContext` that:

1. On authenticated mount, calls `POST /api/v1/permissions/check` for the current user's full permission set and caches results via RTK Query (TTL: session lifetime, re-fetched on `rbac:invalidated` SignalR event)
2. Exposes a `useAbility()` hook returning `can(action: string, resource: string): boolean`
3. Wraps all mutation-capable UI elements in an `<Authorized action="..." resource="...">` component that either disables or hides the element based on the result

```typescript
// Usage examples
<Authorized action="role:create" resource="roles">
  <Button onClick={openCreateModal}>New role</Button>
</Authorized>

<Authorized action="tenant:create" resource="tenants" fallback={null}>
  <TenantCreateButton />
</Authorized>
```

The `fallback` prop controls whether the element is hidden (`null`) or rendered as disabled (default). Document when to use each.

**Critical rule**: UI-level RBAC is a UX convenience only — the backend enforces all rules authoritatively. Never skip a backend call because the UI thinks the user lacks permission. The backend will reject unauthorised requests regardless.

## Input Validation & XSS Prevention

* All form inputs validated with Zod schemas before submission via React Hook Form's `resolver`. Define schemas in `/src/features/{feature}/types.ts` alongside TypeScript types.
* Never use `dangerouslySetInnerHTML`. If rich text is required (policy description), use a sanitised renderer (DOMPurify).
* All query parameters and route params validated with Zod before use. Never trust URL values directly.
* Content-Security-Policy headers are set by the backend/CDN, but the React app must not inject inline scripts or dynamic `eval` that would violate a strict CSP.
* Provide example Zod schemas for: `CreateRoleInput`, `AssignRoleInput`, `CreatePolicyInput`, `CreateDelegationInput`.

# 🧭 Phase 5 — UX & Usability

## Loading States

Every RTK Query-backed view must handle all three states explicitly — no component may render without addressing loading and error:

* **Loading** — show a `<Skeleton>` that matches the shape of the loaded content (table skeleton, card skeleton, etc.). Never show a full-page spinner for partial data loads.
